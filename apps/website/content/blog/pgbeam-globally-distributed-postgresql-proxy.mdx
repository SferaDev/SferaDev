Every company running PostgreSQL across regions faces the same problem: latency. A database in Virginia serves a request from Tokyo in 100-200ms per round trip. But a new PostgreSQL connection is not one round trip. It is four: TCP handshake, TLS negotiation, PG startup, and authentication. That is 400-800ms before the first query executes.

In serverless and edge architectures, where connections are short-lived, teams pay this cost on nearly every request. The result is either degraded user experience or expensive over-provisioning of read replicas, along with the operational complexity that comes with them: replication lag, connection routing, and ORM-level read/write splitting.

## The market gap

The solutions available today are either incomplete or locked to a specific platform:

- **PgBouncer** handles connection pooling but offers no query caching. Every read still round-trips to the origin.
- **Cloudflare Hyperdrive** provides pooling and caching, but only works from Cloudflare Workers, locking teams into one compute platform.
- **Prisma Accelerate** offers the same capabilities, but requires the Prisma ORM, locking teams into one data layer.
- **RDS Proxy** is VPC-bound, single-region, AWS-only, and adds no caching.

There is no platform-agnostic, managed solution that combines connection pooling and query caching across multiple regions. PgBeam fills that gap.

## What PgBeam does

PgBeam is a managed PostgreSQL proxy deployed across six global regions (Virginia, Oregon, Dublin, Mumbai, Singapore, Tokyo). It speaks the PG wire protocol natively. Every PostgreSQL driver and ORM works unmodified. Adoption requires a single environment variable change:

```bash
# Before
DATABASE_URL=postgresql://user:pass@rds-instance.us-east-1.rds.amazonaws.com:5432/mydb

# After
DATABASE_URL=postgresql://user:pass@abc.gw.pgbeam.app:5432/mydb
```

No SDK integration. No code changes. No migration. This matters for enterprise adoption: PgBeam slots into existing stacks without engineering effort or vendor risk.

Three capabilities compound to reduce database load and improve response times:

1. **Global routing.** GeoDNS directs each connection to the nearest proxy region, minimizing network hops.

2. **Connection pooling.** Warm upstream connections eliminate the 4-RTT handshake overhead. Pooling alone reduces connection establishment cost by 3-7x, which directly translates to fewer database connections required and lower infrastructure spend.

3. **Edge query caching.** Read query results are cached locally in the nearest region with stale-while-revalidate semantics. Cached responses are served as native PG wire protocol messages, fully transparent to the application. Writes and transactions always pass through to the origin.

## Architecture

PgBeam is built on a control plane / data plane separation:

- **Control plane** (Go API on ECS Fargate): manages tenants, projects, and configuration. Streams config to data planes via gRPC. Handles billing, plan enforcement, and API key management.

- **Data planes** (Go, per region): run the PG wire proxy using pgproto3. Handle TLS/SNI-based tenant routing, connection pooling, and query caching. Each region operates in a dedicated VPC. All six regions are connected via full-mesh VPC peering for inter-region relay over private network.

- **Multi-tenant isolation**: per-project connection limits, rate limits, cache namespaces, and SNI-based routing. Organization-level plan enforcement on resource creation and query throughput.

The proxy validates client credentials against the upstream database on every new connection. PgBeam never stores database passwords. Credential passthrough means zero trust surface area added.

## Known limitations

- **Read caching only.** PgBeam does not replicate data. Writes pass through to the origin.
- **Eventual consistency.** Cached reads may be up to 60 seconds stale by default. TTL is configurable per project. Caching can be disabled entirely.
- **No cross-region cache coordination.** Each region's cache is independent, expiring via TTL.
- **PostgreSQL only.** No MySQL or other database support today.

These are deliberate scope constraints, not technical debt. Read caching with stale-while-revalidate covers a large class of workloads without the complexity of distributed cache invalidation.
